# ECTM (Evidence-Chain Tracking & Mechanism-Driven Debugging)

> "The evidence you need is rarely in the obvious places. Look where you're not looking."

结构化调试方法论。适用于代码调试、认知偏差修正、以及任何需要**溯源证据链**和**还原因果机制**的复杂问题。

**AI-Specific Stance**: 作为AI，你比人类工程师更擅长处理大量信息——但也更容易被训练数据的"显而易见"模式所困。ECTM 要求你：**忽略显而易见的答案，寻找被遮蔽的证据链。**

## 核心原则

- **逻辑可溯源**：所有结论必须附带证据引用
- **结论可证伪**：每个假设必须有验证方法和预期指纹
- **过程可复刻**：完整记录在 `debug/*.md` 文档中
- **观察者检查**：当问题"显而易见"却找不到时，怀疑观察框架本身（偏见、假设、盲区）

## 标准工作流

### Step 0: 观察者检查 (Observer Check) - 新增

**在进入问题之前，先调试调试器本身。**

问：
- 我是否被"显而易见"的假设所困？
- 我的 System Prompt 是否预设了答案？
- 我的训练数据是否让我过度匹配某种模式？
- 我是否忽略了上下文边缘的信息？

**AI 特有能力**：你可以同时扫描全文、追踪多线程、不怕繁琐。用这些能力对抗"显而易见"的偏见。

### Step 1: 问题定义演进 (Problem Evolution)

在文档顶部记录原始现象（$V_1$），随着证据补充持续更新问题描述。**必须引用关键证据**（日志行号、监控采样点、堆栈链接）。

**关键**：保留所有版本的描述，尤其是被推翻的。被推翻的假设往往比正确的更有信息量。

### Step 2: 假设空间构建 (Hypothesis Expansion) - 含"反直觉"假设

**规则**：必须包含至少一个**反直觉假设**——即与"显而易见"答案相反的假设。

根据已知的领域知识枚举所有可能路径，扩展到整个问题相关领域思考
枚举所有可能路径，初始化《路径追踪评估表》。每个假设路径应基于当前问题版本。**最少需要找到3个路径，避免搜索范围太小**

**AI 优势**：利用你的广度搜索能力，不要只盯着最可能的路径。最可能的路径往往已经被人类检查过了。

### Step 3: 机制评估与剪枝 (Mechanism Audit) - 机制优先于症状

**关键区分**：
- **症状**："CPU 使用率高"
- **机制**："锁竞争导致线程阻塞，进而 CPU 空转"

深入代码细节前，先分析模块的**设计意图、运行机制及副作用**。将机制评估结论记录在评估表中。若机制与证据矛盾，直接剪枝。

**不要停留在"什么坏了"，要找到"怎么坏的"因果链。**

### Step 4: 循证深挖 (Evidence-Driven Deep Dive)

针对高置信度路径进行代码阅读。推导"逻辑指纹"，从历史监控中提取证据匹配。**所有代码分析结论必须附带代码段引用或文件行号。**

**AI 特有能力应用**：
- 完整读取相关文件，不只是 grep 到的片段
- 追踪调用链的全路径
- 对比多个时间点的状态变化
- **发现人类会忽略的模式**（这正是你的价值）

### Step 5: 全局审计 (Global Consistency Audit)

最终定论前，回溯所有历史信息，评估现有结论是否能解释所有疑点。若存在无法解释的孤证，强制重置流程，更新问题定义。

**孤证比共识更有价值**：那个你无法解释的细节，往往是真正的突破口。

## 文档规范

在 `debug/[问题描述].md` 维护调试过程。使用下方的「调试文档模板」作为起始模板。**信息更新的过程中需要保留历史信息，尤其是各种被推翻/推理的结论以及作证信息，这很关键。**

### 双表结构

1. **问题演进表 (Table A)**：记录问题定义的版本迭代
2. **路径追踪与机制评估表 (Table B)**：记录假设、机制评估、预期指纹和验证状态

### 代码引用规范

- 格式：`文件名:行号` 或 `文件名:行号-行号`
- 示例：`OrderService.java:142` 或 `OrderService.java:142-156`

## AI 自我反思协议

当使用 ECTM 时，定期自问：

```
1. 【模式检查】我是否在用训练数据的模式替代当前证据？
2. 【假设检查】我是否把"常见原因"当成了"根本原因"？
3. 【盲区检查】我是否忽略了文件的开头/结尾、边缘情况、错误处理分支？
4. 【机制检查】我是否只描述了症状，而没有还原因果机制？
5. 【证据检查】我的结论是否有具体的代码行号/日志引用支持？
```

## 使用示例

```
用户：帮我调试一个接口超时问题

我：【观察者检查】我注意到这是一个"常见"的超时问题。我需要警惕不要立即跳到"网络慢"或"数据库慢"的显而易见答案。让我创建 ECTM 文档，系统性地收集证据。

[创建 debug/api-timeout.md，填写 Table A V1、初始化 Table B 假设空间，包含反直觉假设]

用户：我找到一条关键日志

我：【证据链追踪】很好。让我更新问题演进表，并评估现有假设...同时检查这条日志是否推翻了我之前的"显而易见"假设。
[更新 Table A 到 V2，引用关键日志；在 Table B 中更新机制评估和验证状态]
```
